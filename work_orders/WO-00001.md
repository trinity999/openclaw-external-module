## CONTEXT
We are in Day-1 controlled production mode running a hybrid recon stack. Current mission requires high-throughput scanning across very large datasets while preserving strict data integrity and low operational noise.

## OBJECTIVE
Produce a deterministic orchestration architecture for parallel recon execution (DNS/HTTP/enrichment/ingest/validation) that maximizes throughput without corrupting graph/columnar data stores.

## SYSTEM OVERVIEW SNAPSHOT
- Architecture: controller-brain + remote-worker-muscle model.
- Compute topology:
  - Controller node (Hostinger VPS): orchestration, linking, ingest, aggregation.
  - Oracle worker node: heavy scan execution (dnsx/httpx/nuclei/katana/etc.).
- Orchestration model: queue-driven, checkpointed, multi-lane execution.
- Data flow: root domains -> subdomains -> DNS -> HTTP -> enrich -> Neo4j relationships + ClickHouse time-series.
- Operational phase: Day-1 activation (stability-first, controlled autonomy).

## CURRENT STATE OF PRODUCT
- Active modules: KRIL, USIL, MACFL, AWSEM scheduler, ORS, ECE.
- Mission control: Mattermost-first; Discord fallback.
- Knowledge layer: Obsidian + Notion hourly sync.
- Known limitations:
  - Large-scale lane tuning not yet benchmarked in production load.
  - Current first-run chunking started; full 10M+ workflow not stress-validated.
- Experimental components: hybrid lane auto-tuning heuristics.
- Active assumptions:
  - Oracle worker handles majority heavy scans.
  - Controller remains below saturation for ingest/control.

## SCALE CHARACTERISTICS
- Input sizes:
  - roots: ~1.6k
  - subdomains: ~10,011,956
- Growth expectation: steady accumulation from ongoing recon.
- Concurrency expectation:
  - Oracle: medium-high parallelism
  - Controller: low-medium parallelism
- Throughput target: sustained multi-lane processing with no integrity regressions.
- Latency tolerance: moderate; correctness > single-pass speed.

## CONSTRAINTS
- JSON-first outputs mandatory.
- Ingestion must be idempotent.
- No destructive mutation of source datasets.
- No privilege escalation / unsafe remote ops.
- Preserve checkpoint replayability.
- Keep token/cost discipline (reuse knowledge before expensive reasoning).

## KNOWN RISKS
- Parallel scans may create duplicate/contradictory writes.
- Queue imbalance may starve one node or overload controller.
- DNS/HTTP race conditions may break linking consistency.
- Retry loops may amplify cost and noise.
- Mis-keyed dedup contracts may hide or duplicate findings.

- Auditability and replayability over opaque speed.
- Controlled parallelism over uncontrolled concurrency spikes.
- Stability first; optimization second.
